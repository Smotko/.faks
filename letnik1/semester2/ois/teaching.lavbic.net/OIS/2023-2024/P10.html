17.1 Objektni razvoj

17.1.1 RUP proces razvoja

Objektni proces razvoja je skupinski pristop k razvoju, ki temelji na poenotenem procesu (angl. Rational Unified Process (RUP)) in jeziku za modeliranje (angl. Unified Modeling Language (UML)), kot prikazuje slika 17.1.

Slika 17.1: Objektni razvoj

RUP je primer procesa objektnega razvoja, katerega avtor je podjetje Rational Software in ima opredelje faze življenjskega cikla, ki se izvajajo v iteracijah, kot to prikazuje slika 17.2.

Slika 17.2: Faze življenjskega cikla, iteracije in postopki RUP procesa razvoja

RUP zajema 4 faze življenjskega cikla (glej sliki 17.2 in 17.3):

  • začetna faza - vzpostavitev projekta, opredelitev okvirjev obravnavanega področja, načrtovanje virov,
  • zbiranje informacij - zbiranje informacij o obravnavanem področju, specifikacija značilnosti, načrtovanje arhitekture,
  • konstrukcija - konstrukcija izdelka,
  • prevzem - predaja izdelka v uporabo končnemu uporabniku.

Slika 17.3: Faze življenjskega cikla in mejniki

RUP predvideva uporabo različnih UML diagramskih tehnik za specifične potrebe, kot to prikazuje slika 17.4.

Slika 17.4: UML diagramske tehnike

Glavne lastnosti RUP procesa razvoja:

  • daje smernice za učinkovit razvoj kakovostne programske opreme,
  • zmanjšuje tveganja in povečuje predvidljivost,
  • zajema in vpeljuje najboljše izkušnje,
    • učenje iz izkušenj drugih,
    • mentorstvo v elektronski obliki,
    • razširitev izobraževalnega gradiva,
  • pospešuje skupno vizijo in kulturo,
  • vpeljuje načrt za uvedbo orodij,
  • omogoča enostaven in hiter dostop do informacij v elektronski obliki (npr. spletna stran ipd.).

Primeri uporabe (angl. Use cases) so hrbtenica procesa RUP (glej sliko 17.5).

Slika 17.5: Vloga primerov uporabe

Pri gradnji modela sistema imajo primeri uporabe zelo pomembno vlogo, saj na podlagi njih začnemo gradnjo modela sistema po posameznih fazah življenjskega cikla (glej sliko 17.6).

Slika 17.6: Gradnja modela sistema

Prednosti arhitekturno usmerjenega pristopa RUP so prikazane na sliki 17.7.

Slika 17.7: Prednosti RUP procesa razvoja

17.1.1.1 Zajem zahtev

Slika 17.8: Zajem zahtev po fazah življenjskega cikla

Namen zajema zahtev je:

  • doseči soglasje s stranko oz. uporabniki, kaj naj sistem dela,
  • omogočiti razvijalcem boljše razumevanje zahtev sistema,
  • določiti meje sistema,
  • zagotoviti osnovo za načrtovanje tehnične vsebine,
  • definirati grafični uporabniški vmesnik (angl. Graphical User Interface (GUI)) sistema.

Namen zajema zahtev je podoben kot pri strukturnem pristopu. V določeni meri uporabljamo tudi podobne tehnike kot pri strukturnem pristopu.

RUP za zapis zahtev uvaja višjo stopnjo formalnosti (glej sliko 17.4):

  • primeri uporabe (jezik UML),
  • opis s tokovi dogodkov,
  • uporaba drugih tehnik UML v posebnih primerih (npr. diagrami stanj, diagrami aktivnosti, diagrami interakcije).

17.1.1.2 Analiza in načrtovanje

Slika 17.9: Analiza in načrtovanje po fazah življenjskega cikla

Namen analize in načrtovanja je:

  • pretvoriti zahteve v načrt sistema,
  • razviti robustno arhitekturo sistema,
  • prilagoditi načrt izvedbenemu okolju.

Povezava z drugimi postopki:

  • poslovno modeliranje postavi organizacijski kontekst sistema,
  • postopek zajema zahtev je primarni vhod v postopek analize in načrtovanja,
  • postopek izvedbe realizira sistem na osnovi načrta.

Univerzalni modelirni jezik (UML) je jezik za specifikacijo, vizualizacijo, konstrukcijo in dokumentacijo izdelkov v okviru objektnega razvoja informacijskih rešitev.

17.1.1.3.1 Diagrami primerov uporabe

Diagrami primerov uporabe (DPU) (angl. Use Case Diagrams (UCD)) predstavljajo povezavo med akterji (uporabniki sistema) in procesi, ki v sistemu tečejo. Namen uporabe DPU je predstavitev in zajemanje uporabniških zahtev, s katerimi modeliramo tudi vplive okolja na sistem. Sistem modeliramo kot množico procesov, ki uporabniku ponujajo usluge sistema, po drugi strani je to množica akterjev, ki vplivajo na sistem.

Gradniki:

  • primer uporabe (angl. use case),
  • akter (angl. actor) in
  • povezave med primeri uporabe in akterji.

Primer uporabe opisuje posamezen tok dogodkov skozi sistem, ki akterju vrne merljiv rezultat (npr. dvig denarja na bankomatu na sliki 17.14).

17.1.1.3.5 Diagram zaporedja

UML diagrami zaporedja (angl. UML sequence diagrams) so še ena izmed pomembnih načrtovalskih tehnik. Ponavadi se uporablja pri načrtovanju, preden razvojna ekipa začne s programiranjem. Diagrami zaporedja se uporabljajo za prikaz, kako objekti v programu medsebojno sodelujejo pri izvedbi določenih nalog. Diagram zaporedja predstavlja zemljevid komunikacije med različnimi akterji, kjer so označena sporočila, poslane od akterja do akterja.

Diagram zaporedja pomaga vizualizirati razrede, ki jih sistem kreira ter funkcije, ki jih bomo morali implementirati. Takšen pogled lahko tudi ponazori morebitne težave v sistemu, ki prej niso bile znane.

Poglejmo si posamezne komponene diagrama zaporedja:

  • Škatle predstavljajo vloge, ki jih imajo posamezni objekti. Vloga (angl. Role) je običajno poimenovana po razredu objekta.
  • Življenjske črte (angl. lifelines) so navpične črtkane črte in se v diagramu uporabljajo za označevanje objekta v času.
  • Puščie s polno črto se uporabljajo za prikaz sporočil (angl. messages), ki so poslana od enega objekta do drugega ali od pošiljatelja do prejemnika. Sprejemniki so koncu usmerjene puščice.
  • Puščice s črtkano črto se uporabljajo za prikaz vrnitve podatkov in nadzora nazaj začetnim objektom. Kratek opis vrnitve podatkov ali nadzor je vključen v obliki besedila nad puščico.
  • Majhni pravokotniki vzdolž življenjske črte predmeta označujejo aktivacijo metode. Objekt aktiviramo vedno, ko objekt pošlje, prejme ali čaka na sporočilo.
  • V diagram zaporedja so lahko vključeni tudi akterji (angl. actors), če uporabljajo objekte ali z njimi komunicirajo. Akterji so ponavadi prikazani z ikono človečka.

Diagram zaporedja ponavadi uokvirimo, kjer je naslov diagrama naveden v zgornjem levem kotu (glej sliko 17.30). Dobra praksa je, da zagotovimo smiseln naslov, saj se bo diagram uporablja kot referenca pri razvoju. Druga dobra praksa je risanje objektov od leve proti desni v zaporedju medsebojnega sodelovanja.

17.1.1.3.6 Diagram stanj

UML diagrami stanj (angl. UML state diagrams) so druga vrsta diagramov, ki se uporablja za opis, kakose sistemi obnašajo in odzivajo. Sledijo stanjem sistema ali posameznega objekta in prikazujejo spremembe med stanji, ko se v sistemu zgodi vrsta dogodkov.

Diagram stanja ponazarja vedenja objekta s prikazom spreminjajočih se stanj objekta. Te spremembe so odziv na različne dogodke. Stanje (angl. State) je način, kako objekt obstaja v določenem trenutku. Stanje objekta je določeno z vrednostmi njegovih atributov.

Dobra analogija stanjem je primer avta. Avto s samodejnim menjalnikom ima lahko različna stanja: parkiraj, vzvratno, nevtralno in vožnja. Če je avto v stanju vzvratno, se obnaša tako, da se premika nazaj. Če se želite premakniti naprej, morate spremeniti stanje avta v vožnja. To je podobno stanju objektov v sistemu. Ko je objekt v določenem stanju, se obnaša na določen način ali ima atribute nastavljene na določene vrednosti. Elementi diagrama stanja so:

  • Poln krog označuje začetno stanje objekta. Vsak diagram stanja se začne s polnim krogom.
  • Zaobljen pravokotnik označuje druga stanja. Ti pravokotniki imajo tri razdelke: ime stanja, spremenljivke stanja in aktivnosti.
    • Imena stanj morajo biti kratki, smiselni naslovi stanj objekta. Vsako stanje mora imeti vsa ime stanja.
    • Spremenljivke stanja so podatki, pomembni za stanje objekta.
    • Aktivnosti (angl. Activities) so akcije, ki se izvajajo v določenem stanju. Za vsako stanje poznamo tri vrste aktivnosti: Entry, Exit in Do. Aktivnosti Entry so tiste, ki se zgodijo, ko se v stanje vstopi iz drugega stanja. Aktivnosti Exit so tiste, ki se zgodijo, ko se stanje zapusti in se zgodi prehod v drugo stanje. Aktivnosti Do se zgodijo, ko je predmet v določenem stanju.
  • Puščie označujejo prehode (angl. transitions) iz enega stanja v drugo. Prehode običajno sproži dogodek, ki je zapisan nad puščico.
    • Vsaka puščica prehoda ima vedno opredeljen dogodek, prisoten pa je lahko tudi pogoej in akcija. Prehod in akcija se zgodi iz danega stanja, če se dogodek zgodi in je pogoj resničen.
  • Krog z zapolnjenim notranjim krogom označuje konec, ki predstavlja uničenje objekta ali zaključek procesa. Vsi diagrami nimajo zaključka, saj lahko delujejo neprekinjeno.

Diagrami stanj so lahko koristni pri določanju dogodkov, ki se lahko zgodijo med življenjsko dobo objekta, kot so npr. različni uporabniški vnosi, in kako naj se objekt obnaša, ko do teh dogodkov pride, kot je preverjanje pogojev in izvajanje akcij. Včasih je morda lažje videti spremembo stanja v diagramu, namesto branja izvorne kode.

Diagrami stanj lahko pomagajo tudi prepoznavati težave v sistemu, kot je odkrivanje stanj, ki niso načrtovana. Pomagajo lahko tudi pri pripravi testov - poznavanje različnih stanj sistema lahko pomaga pri zagotavljanju popolnosti in pravilnosti testov.

17.2.1 Predstavitev problema

Celovit postopek razvoja informacijske rešitve bomo izvedli s pomočjo OO pristopa, in sicer v naslednjih korakih:

  • zbiranje zahtev (glej poglavje 17.2.2),
  • analiza (identifikacija primerov uporabe, podrobnih tokov dogodkov, opredelitev funkcionalnih testov in analiza razredov) (glej poglavje 17.2.3),
  • načrt
    • pregledni načrt (dodelitev odgovornosti razredom, razredni diagrami, diagrami stanj in sodelovanja) (glej poglavje 17.2.4) in
    • podroben načrt (atributi, metode, paketi) (glej poglavje 17.2.5) ter
  • implementacija (Java) (glej poglavje 17.2.6).

17.2.2 Zajem zahtev

Razvita programska oprema bo skrbela za delovanje bankomata, ki je sestavljen iz naslednjih komponent:

  • bralca bančnih kartic,
  • tipkovnice in zaslona za komunikacijo s stranko,
  • predala za oddajo kuvert,
  • reže za denar,
  • tiskalnika za tiskanje potrdil in
  • varnostnega stikala za zagon oz. zaustavitev naprave s strani operaterja.

Bankomat lahko hkrati uporablja ena stranka. Stranka ob začetku interakcije vstavi bančno kartico in vnese številko PIN, kjer se oba podatka nato posredujeta banki v preverjanje. Stranka lahko nato izvede eno ali več transakcij. Kartica ostane v bankomatu, dokler stranka ne želi opraviti nobene transakcije več.

Bankomat mora stranki nuditi naslednje storitve:

  • Stranka lahko dvigne denar iz poljubnega računa, ki je povezan z bančno kartico, v bankovcih vrednosti večkratnika 10 €.
  • Stranka lahko položi denar na poljuben račun, ki je povezan z bančno kartico, glede na denar oz. čeke v kuverti. Stranka vnese višino depozita, ki se nato ročno preveri s strani operaterja bankomata. Potrditev razpoložljivosti računa za depozit mora biti s strani banke potrjena pred sprejetjem kuverte.
  • Stranka lahko izvede prenos med poljubnima računoma, povezanima z vstavljeno bančno kartico.
  • Stranka lahko zahteva izpis stanja na poljubnem računu, povezanem z vstavljeno bančno kartico.

Stranka lahko v vsakem trenutku prekine transakcijo s pritiskom na gumb Prekliči. Bankomat pri vsaki transakciji komunicira z banko in pridobi potrditev izvedljivosti transakcije. Transakcija je namreč zaključena, ko jo potrdi banka. V primeru depozita se posreduje banki še dodatno sporočilo, da je stranka na bankomatu oddala kuverto. Če stranka v predpisanem časovnem obdobju kuverte z denarjem ne odda ali pritisne tipko Prekliči, se dodatnega sporočila banki ne posreduje.

Če banka ugotovi, da je PIN številka napačna, jo mora stranka vnesti ponovno, da lahko uspešno izvede transakcijo. V primeru, da stranka 3x napačno vnese PIN, bankomat zadrži bančno kartico in stranka mora kontaktirati svojo banko. Če transakcija ni uspešna zaradi razloga, ki ni napačna PIN številka, bo bankomat obvestilo o napaki izpisal na zaslon in vprašal stranko, ali želi izvesti še kakšno transakcijo.

Bankomat stranki omogoči izpis potrdila o opravljeni transakciji, ki vsebuje datum, uro, lokacijo bankomata, tip transakcije, račun, znesek in končno stanje na računu.

Bankomat ima na voljo varnostno stikalo za vklop oz. izklop. Ko operater vklopi bankomat, mora preveriti točen znesek denarja, ki je na voljo. Bankomat se lahko izklopi le takrat, ko ga stranka ne uporablja. Ko je bankomat izklopljen, lahko operater odstrani kuverte z depoziti in ponovno naloži denar ter prazen papir za izpis potrdil.

Bankomat hrani tudi svoj interni dnevnik transakcij za lažje obvladovanje morebitnih napak, do katerih lahko pride zaradi strojne napake med transakcijo. V dnevnik se beležijo vsi zagoni in ustavitve bankomata ter vsako poslano sporočilo banki (vključno z odgovorom). Dnevniški zapisi vsebujejo podatke o številki kartice in zneskih, vendar zaradi varnostnih razlogov ni nikoli podatka o PIN številki.

17.2.3.4 Opis PU3: Interakcija

Slika 17.35: PU3: Interakcija PlantUML

  • Do interakcije pride, ko stranka vstavi kartico v bralnik kartic bankomata.
  • Bankomat kartico povleče in jo prebere.
    • Če bralnik kartice ne more prebrati, jo izvrže in na zaslonu izpiše obvestilo o napaki.
  • Bankomat od stranke zahteva vnos PIN številke in šele takrat ji je dovoljeno izvajati transakcije, ki jih izbere iz menija.
  • Po vsaki transakciji stranka izbere, ali želi izvesti še kakšno transakcijo.
  • Ko stranka zaključi s transakcijami, bankomat kartico izvrže in jo vrne stranki ter na ta način zaključi interakcijo.
  • Če je transakcija prekinjena zaradi prevelikega števila napačno vnesenih PIN številk, se interakcija ravno tako zaključi, vendar kartica ostane v bankomatu.
  • Stranka lahko med interakcijo le-to prekine s pritiskom na gumb Prekliči.

17.2.3.5 Opis PU4: Izvedba transakcije

Slika 17.36: PU4: Izvedba transakcije PlantUML

  • Izvedba transakcije je abstraktna generalizacija. Vsak specifičen konkreten tip implementacije vsebuje svoje operacije. Tok dogodkov je podan na tem mestu in opisuje skupno delovanje.
  • Izvedba transakcije se požene znotraj interakcije, ko stranka izbere tip transakcije iz menija. V nadaljevanju stranka poda ustrezne podrobnosti (npr. željen račun, znesek itd.). Transakcija je nato posredovana banki skupaj s potrebnimi informacijami o kartici in številki PIN.
  • Če banka transakcijo potrdi, se izvedejo še ustrezne akcije (npr. oddaja denarja ali polog s položnico), nato pa se natisne še potrdilo o transakciji. Zatem se stranka odloči, ali želi izvesti še kakšno transakcijo.
  • Če banka sporoči, da je številka PIN napačna, se izvede drug primer uporabe in se nato morebiti vrne izvajanje nazaj v ta primer uporabe. Če bankomat kartico zadrži zaradi prevelikega števila napačnih vnosov PIN številke, stranka ne bo mogla več izbirati akcij.
  • Če stranka transakcijo prekliče ali če je neuspešna zaradi drugega razloga kot napačen PIN, se na zaslonu prikaže opis napake in stranki se ponudi, da izbere drugo transakcijo.
  • Stranka lahko prekliče transakcijo s klikom na gumb Prekliči.
  • Vsa sporočila med banko in bankomatom v obe smeri so shranjena v internem dnevniku bankomata.

17.2.3.6 Opis PU5: Dvig denarja

Slika 17.37: PU5: Dvig denarja PlantUML

Sistem preveri, ali je v bankomatu dovolj denarja, preden pošlje transakcijo v obdelavo na banko. Če denarja ni dovolj, potem sistem stranko obvesti, naj izbere drug znesek. Če je transakcija potrjena s strani banke, se stranki izda ustrezen znesek gotovine in natisne potrdilo. Izdaja denarja se shrani v interni dnevnik bankomata.
  • Stranka lahko transakcijo prekliče s pritiskom na gumb Prekliči.

Slika 17.38: PU6: Polog denarja PlantUML

  • Pri pologu denarja mora stranka izbrati račun, na katerega želi položiti denar, in višino depozita. V okviru transakcije se pri banki preveri, ali lahko bankomat opravi depozit na izbran račun. Če je transakcija potrjena, bankomat sprejme kuverto z denarjem ali čeki ter nato izda potrdilo. Ko je kuverta sprejeta, se posreduje še eno obvestilo banki, kjer se potrdi prenos denarja na račun, seveda po ročnem preverjanju vsebine kuverte. Omenjena komunikacija se hrani tudi v internem dnevniku bankomata.
  • Depozit lahko stranka prekliče s pritiskom na gumb Prekliči vsak trenutek pred oddajo kuverte. Prav tako se transakcija prekliče, če stranka ne odda kuverte z denarjem v določenem časovnem obdobju.

17.2.3.8 Opis PU7: Prenos denarja

Slika 17.39: PU7: Prenos denarja PlantUML

  • Pri prenosu denarja mora stranka izmed vseh možnih računov izbrati tistega, iz katerega denar prenaša, in drug račun, na katerega denar prenaša, ter vsoto. Če banka potrdi transakcijo, bankomat natisne potrdilo. Omenjena komunikacija se hrani tudi v internem dnevniku bankomata.
  • Prenos denarja lahko stranka prekine s pritiskom na tipko Prekliči.

17.2.3.9 Opis PU8: Pregled stanja

Slika 17.40: PU8: Pregled stanja PlantUML

  • Stranka iz menija izbere pregled stanja za posamezen račun. Ko banka vrne stanje, se izpiše potrdilo. Omenjena komunikacija se hrani tudi v internem dnevniku bankomata.
  • Stranka lahko akcijo vedno prekliče s klikom na gumb Prekliči.

17.2.3.10 Opis PU9: Napačen PIN

Slika 17.41: PU9: Napačen PIN PlantUML

  • PU se začne izvajati, ko pri poskusu izvedbe transakcije banka sporoči, da je stranka vnesla napačno PIN številko.
  • Stranka mora nato ponovno vnesti PIN in prvotna transakcija se zopet posreduje banki v potrditev.
    • Če banka v tem trenutku transakcijo potrdi ali jo zavrne zaradi kakšnega drugega razloga, se osnovni PU nadaljuje, sicer se postopek ponovnega vnosa PIN številke ponovi.
  • Ko je enkrat PIN številka pravilno zajeta, se uporablja za trenutno transakcijo in vse nadaljnje v okviru interakcije.
  • Če stranka 3x napačno vnese PIN številko, bankomat obdrži kartico in na zaslonu se izpiše napaka ter priporočilo, naj stranka kontaktira svojo banko. S tem se zaključi tudi interakcija.
  • Če stranka med tem procesom pritisne gumb Prekliči, se transakcija prekliče.

package bankomatIs;

import java.net.String;
import bankomatIs.fizicno.*;

public class Bankomat implements Runnable {

  public Bankomat (int id, String lokacija, String nazivBanke, String naslovBanke) {
    this.id = id;
    this.lokacija = lokacija;
    this.nazivBanke = nazivBanke;
    this.naslovBanke = naslovBanke;

    dnevnik = new Dnevnik();
    bralnikKartic = new BralnikKartic(this);
    rezaZaDenar = new RezaZaDenar(dnevnik);
    uporabniskaPlosca = new UporabniskaPlosca();
    povezavaDoBanke = new PovezavaDoBanke(dnevnik, naslovBanke);
    nadzornaPlosca = new NadzornaPlosca(this);
    tiskalnikPotrdil = new TiskalnikPotrdil();  

    stanje = STANJE_UGASNJEN;
    vklopljen = false;
    karticaVstavljena = false;
  }
  
  public void run() {
    Interakcija trenutnaInterakcija = null;

    while (true) {
      switch (stanje) {
        case STANJE_UGASNJEN:
          uporabniskaPlosca.prikaziObvestilo (”Trenutno ni na voljo");
          synchronized (this) {
            try {
              wait();
            } catch (InterruptedException e) { }
          }
          if (vklopljen) {
            izvediZagon ();
            stanje = STANJE_V_MIROVANJU;
          }
          break;
        case STANJE_V_MIROVANJU:
          uporabniskaPlosa.prikaziObvestilo (”Prosim vstavite kartico");
          karticaVstavljena = false;
          synchronized (this) {
            try { 
              wait();
            } catch (InterruptedException e) { }
          }      
          if (karticaVstavljena) {
            trenutnaInterakcija = new Interakcija (this);
            stanje = STANJE_IZVAJA_STORITEV_ZA_STRANKO;
          } else if (! vklopljen) {
            izvediIzklop();
            stanje = STANJE_UGASNJEN;
          }
          break;
        case STANJE_IZVAJA_STORITEV_ZA_STRANKO:
          trenutnaInterakcija.kreirajInterakcijo ();
          status = STANJE_V_MIROVANJU;
          break;
      }
    }
  }

    vklopljen = true;
    notify();
  }

    vklopljen = false;
    notify();

package bankomatIs;

import banka.Kartica;
import banka.Status;
import banka.Potrdilo;
import bankomatIs.fizicno.UporabniskaPlosca;
import bankomatIs.transakcije.Transakcija;

public class Interakcija {

  public Interakcija (Bankomat bankomat) {
    this.bankomat = bankomat;
    stanje = STANJE_BRANJE_KARTICE;
  }

  public void kreirajInterakcijo () {
    Kartica kartica = null;
    Transakcija trenutnaTransakcija = null;
    while (stanje != STANJE_KONCNO) {
      switch (stanje) {
        case STANJE_BRANJE_KARTICE:
          kartica = bankomat.vrniBralnikKartic().preberiKartico();
          if (kartica != null)
            stanje = STANJE_BRANJE_PIN_STEVILKE;
          else {
            bankomat.vrniUporabniskoPlosco().prikaziObvestilo(”Kartice ni mogoče prebrati");
            stanje = STANJE_VRNITEV_KARTICE;
          }
          break;
        case STANJE_BRANJE_PIN_STEVILKE:
          try {
            pin = bankomat.vrniUporabniskoPlosco().zahtevajPIN(”Vnesite svojo PIN številko in pritisnite ENTER. ");
            stanje = STANJE_IZBIRA_TRANSAKCIJE;
          } catch (UporabniskaPlosca.Preklicano e) {
            stanje = STANJE_VRNITEV_KARTICE;
          }
          break;
        case STANJE_IZBIRA_TRANSAKCIJE:
          try {
            trenutnaTransakcija = Transakcija.izvediTransakcijo (bankomat, this, kartica, pin);
            stanje = STANJE_IZVEDBA_TRANSAKCIJE;
          } catch (UporabniskaPlosca.Preklicano e) {
            stanje = STANJE_VRNITEV_KARTICE;
          }
          break;
        case STANJE_IZVEDBA_TRANSAKCIJE:
          try {
            boolean ponovi = trenutnaTransakcija.pozeniTransakcijo ();
            if (ponovi)
              stanje = STANJE_IZBIRA_TRANSAKCIJE;
            else
              stanje = STANJE_VRNITEV_KARTICE;
          } catch (Transakcija.KarticaZasezena e) {
            stanje = STANJE_KONCNO;
          }
          break;
        case STANJE_VRNITEV_KARTICE:
          bankomat.vrniBralnikKartic().vrniKartico();
          stanje = STANJE_KONCNO;
          break;
      }
    }
  }
  
  public void nastaviPIN(int pin) {
    this.pin = pin;
  }

  private Bankomat bankomat;
  private int pin;
  private int stanje;
  private static final int STANJE_BRANJE_KARTICE = 1;
  private static final int STANJE_BRANJE_PIN_ŠTEVILKE = 2;
  private static final int STANJE_IZBIRA_TRANSAKCIJE = 3;
  private static final int STANJE_IZVEDBA_TRANSAKCIJE = 4;

17.3.1 Pravila

  • Cilj:
    • Izdelati vse primere uporabe v čim krajšem času in s čim nižjimi stroški.
  • Omejitve časa in stroškov projekta:
    • maksimalno trajanje je 31 tednov,
    • maksimalen proračun je 100.000 €.
  • Stroški in produktivnost:
    • 1 zaposleni stane 1.000 €/teden,
    • 1 zaposleni izdela 5 enot primerov uporabe/teden (ena enota je enaka relativni ceni izdelave primera uporabe),
    • + 1 teden/iteracijo za načrtovanje in režijo.
  • Dodatne omejitve:
    • relativne cene primerov uporabe (enote) so konstante, pravilne in se ne spreminjajo, razen če na to eksplicitno vpliva nepričakovan dogodek,
    • vsak primer uporabe izdelate v celoti v eni iteraciji,
    • zaposlene je mogoče dodeliti tudi le delno, kar pomeni, da zaposleni neko vlogo opravlja le del časa, kjer je najmanjša enota za delitev 0,1 (10% časa),
    • načrtovani urnik po izračunu morda ne bo celo število; v tem primeru kot rezultat uporabite najbližje celo število, zaokroženo navzgor.
