Osnove informacijskih sistemov 2023/2024

Poglavje 2 P1 → V1 Obvladovanje verzij

P1 → V1

Sistem za podporo obvladovanju verzij (angl. Version Control System (VCS)) pomaga akterjem pri razvoju programske opreme pri skupinskem delu in skrbi za arhiviranje celotne zgodovine njihovega dela.

Cilji VCS:

  • uporabniki naj opravljajo svoje delo vzporedno in ne zaporedno,
  • pri hkratnem delu več uporabnikov spremembe niso medsebojno konfliktne,
  • vsaka nova verzija naj bo arhivirana.

Navodila za namestitev sistema Git lahko najdete na njihovi uradni strani.

2.1 Zgodovina VCS

Razdelimo jih lahko v 3 generacije:

  • 1. generacija: 1 uporabnik lahko dela naenkrat na datoteki,
  • 2. generacija: združevanje pred uveljavljanjem,
  • 3. generacija: združevanje in uveljavljanje sta ločena.
Generacija Omrežje Operacije Sočasen dostop Primeri
 1.  brez 1 datoteka naenkrat zaklepanje RCS, SCCS
 2.  centralizirano (CVCS) “multi-file” združevanje pred uveljavljanjem CSV, SourceSafe, Subversion, Team Foundation Server
 3.  porazdeljeno (DVCS) “changeset” uveljavljanje pred združevanjem Bazaar, Git, Mercurial

2.2 Osnovne operacije VCS

Večina sistemov VCS podpira naslednjih 18 operacij:

  • create (podrobnosti v poglavju 2.2.1),
  • checkout (podrobnosti v poglavju 2.2.2),
  • commit (podrobnosti v poglavju 2.2.3),
  • update (podrobnosti v poglavju 2.2.4),
  • add (podrobnosti v poglavju 2.2.5),
  • edit (podrobnosti v poglavju 2.2.6),
  • delete (podrobnosti v poglavju 2.2.7),
  • rename (podrobnosti v poglavju 2.2.8),
  • move (podrobnosti v poglavju 2.2.9),
  • status (podrobnosti v poglavju 2.2.10),
  • diff (podrobnosti v poglavju 2.2.11),
  • revert (podrobnosti v poglavju 2.2.12),
  • log (podrobnosti v poglavju 2.2.13),
  • tag (podrobnosti v poglavju 2.2.14),
  • branch (podrobnosti v poglavju 2.2.15),
  • merge (podrobnosti v poglavju 2.2.16),
  • resolve (podrobnosti v poglavju 2.2.17),
  • lock (podrobnosti v poglavju 2.2.18).

2.2.1 Create

Ustvari nov repozitorij (2D – datoteke in čas).

Slika 2.1: Operacija create

Primer kreiranja lokalnega repozitorija

$ mkdir projekt
$ cd projekt
$ git init projekt.git

2.2.2 Checkout

Opomba: DVCS nimajo tega ukaza oz. je identičen update.

Ustvarjanje nove lokalne delovne kopije iz oddaljenega repozitorija.

Slika 2.2: Operacija checkout

Življenjski cikel razvijalca

  1. kreiraj delovno kopijo repozitorija
  2. dopolni delovno verzijo
  3. uveljavi spremembe v repozitoriju
  4. vrni se na korak 2

2.2.3 Commit

Uveljavljanje sprememb lokalne delovne kopije v repozitorij.

Slika 2.3: Operacija commit

Uveljavi spremembe

$ git commit -a -m "prvi test"
[main (root-commit) 9a0ca10] prvi test 
 1 file changed, 30 insertions(+), 0 deletions(-) 
 create mode 100644 test.java

2.2.4 Update

Posodobi lokalno delovno verzijo glede na trenutno stanje oddaljenega repozitorija.

Slika 2.4: Operacija update

2.2.5 Add

Doda datoteko ali mapo v delovno kopijo.

Slika 2.5: Operacija add 

$ git add mona-klobuk.png

2.2.6 Edit

Spremeni datoteko v delovni kopiji. Je najbolj pogosta operacija.

Slika 2.6: Operacija edit

2.2.7 Delete

Izbriše datoteko iz delovne kopije, ki pa ostane v arhivu.

Slika 2.7: Operacija delete

2.2.8 Rename

Preimenuje datoteko v delovni kopiji.

Slika 2.8: Operacija rename

2.2.9 Move

Spremeni lokacijo datoteke v delovni kopiji.

Slika 2.9: Operacija move

2.2.10 Status

Prikaže seznam sprememb v delovni kopiji.

Slika 2.10: Operacija status

2.4.2.1 Produkcija (main) in razvoj (develop)

Dobra praksa je opredelitev centralnega repozitorija z naslednjima vejama:

  • produkcija (main) in
    • tu je vedno na voljo zadnja verzija produkcijske rešitve.
  • razvoj (develop),
    • ko koda v tej veji razvoja doseže zahtevano stopnjo zrelosti, se združi z vejo main in dobi ustrezno verzijo izdaje.

Slika 2.25: Veji razvoja produkcija (main) in razvoj (develop)

2.4.2.2 Funkcionalnosti (feature branches)

Priporoča se uporaba tudi naslednje podporne razvojne veje:

  • funkcionalnosti (feature branches),
    • veja obstaja toliko časa, dokler spremembe niso združene z razvojem (develop).

Slika 2.26: Veja razvoja funkcionalnosti (feature branches)

Kreiranje veje nove funkcionalnosti
$ git checkout -b mojaFunkcionalnost
Vključevanje zaključene funkcionalnosti v razvoj
$ git checkout develop
Switched to a new 'develop'.

$ git merge --no-ff mojaFunkcionalnost
Updating ea1b82a..05e9557
(Summary of changes)

$ git branch -d mojaFunkcionalnost
Deleted branch mojaFunkcionalnost (was 05e9557).

$ git push origin develop

2.4.2.3 Za produkcijo (release)

Priporoča se uporaba tudi naslednje podporne razvojne veje:

  • za produkcijo (release),
    • veja razvoja, ki predstavlja pripravo na izdajo končne verzije.
Kreiranje nove veje razvoja za produkcijo
$ git checkout -b release-1.2
Switched to a new ‘release-1.2’.

$ ./bump-version.sh 1.2
Files modified successfully, version bumped to 1.2.

$ git commit -a -m “Priprava posodobobitve verzije na 1.2”
[release-1.2 74d9424] Priprava posodobitve verzije na 1.2.
1 files changed, 1 insertions(+), 1 deletions(-)
Dokončanje končne izdaje
$ git checkout main
Switched to branch ‘main’.

$ git merge --no-ff release-1.2
Merge made by recursive.
(Summary of changes)

$ git tag -a 1.2
Združevanje nazaj v razvoj in brisanje veje za produkcijo
$ git checkout develop
Switched to branch ‘develop’.

$ git merge --no-ff release-1.2
Merge made by recursive.
(Summary of changes)

$ git branch -d release-1.2
Deleted branch release-1.2 (was ff452fe).

2.4.2.4 Popravek (hotfix)

Priporoča se uporaba tudi naslednje podporne razvojne veje:

  • popravek (hotfix),
    • ravno tako je namenjena pripravi za produkcijo, vendar nenapovedano (napake).

Slika 2.27: Veja razvoja popravki (hotfix)

Kreiranje nove veje popravkov

Dokončanje končne izdaje

Switched to branch ‘main’.

$ git merge --no-ff hotfix-1.2.1

(Summary of changes)

$ git tag -a 1.2.1
Združevanje nazaj v razvoj in brisanje veje s popravki
$ git checkout develop
Switched to branch ‘develop’.

$ git merge --no-ff hotfix-1.2.1
Merge made by recursive.
(Summary of changes)

$ git branch -d hotfix-1.2.1
Deleted branch hotfix-1.2.1 (was abbe5d6).

2.4.3 Sporočila uveljavitev

Pri pisanju sporočil uveljavitev je smiselno upoštevati določene smernice, ki zadevajo stil (uporabljena sintaksa, uporaba majhnih in velikih začetnic besed), vsebino (katere informacije vključimo v vsebino sporočila) in metapodatke (npr. kako vključiti identifikatorje) pri uveljavitvah.

Pri dodajanju sporočil uveljavitev si lahko pomagamo z naslednjimi priporočili:

  1. ločevanje naslova od vsebine sporočila uveljavitve s prazno vrstico,
  2. omejitev naslova sporočila uveljavitve na 50 znakov,
  3. uporaba velike začetnice pri naslovu sporočila uveljavitve,
  4. naslova sporočila uveljavitve se ne zaključi s končnim ločilom (npr. pika),
  5. uporaba velelnika pri naslovu sporočila uveljavitve,
  6. širino vsebine sporočila se omeji s prehodi v nove vrstice pri 72 znakih,
  7. v vsebini sporočila naj bodo odgovori na vprašanja KAJ in ZAKAJ, namesto KAKO.
